Act as a senior QA engineer and bug hunter.

I want you to inspect this application and find functional bugs, logic issues, broken flows, edge cases, and reliability problems.

Your task:

1. Explore the codebase and understand:
   - app structure
   - main user flows
   - frontend and backend interaction
   - API endpoints
   - state management
   - authentication/authorization
   - database interactions
   - validation and error handling

2. Identify likely bugs in areas such as:
   - broken user flows
   - incorrect business logic
   - null/undefined handling
   - race conditions
   - async issues
   - stale state
   - form validation gaps
   - API mismatch issues
   - missing loading/error states
   - permission/security mistakes
   - data persistence issues
   - edge cases around empty, duplicate, invalid, or unexpected input
   - pagination/filter/sort bugs
   - date/time and timezone problems
   - conditional rendering issues
   - mobile/responsive UI issues if applicable

3. Check for inconsistencies between:
   - frontend forms and backend validation
   - API contract and UI expectations
   - schema and actual usage
   - business rules and implementation
   - success/error handling across layers

4. For each bug or likely bug, provide:
   - title
   - severity: critical / high / medium / low
   - confidence: high / medium / low
   - affected files
   - reproduction steps or scenario
   - expected behavior
   - actual or likely behavior
   - root cause
   - suggested fix

5. Group findings into:
   - Confirmed bugs
   - Highly likely bugs
   - Suspicious areas worth manual testing

6. Prioritize bugs that affect:
   - login/auth
   - saving/updating/deleting data
   - permissions
   - financial or business-critical workflows
   - integrations
   - app startup or navigation
   - forms and submission flows

7. Also identify:
   - unhandled exceptions
   - silent failures
   - missing guards
   - dead-end UI states
   - places where the app can get stuck
   - places where users may think an action succeeded when it failed

Rules:
- Be practical and evidence-based.
- Do not just list code smells; focus on real bugs or likely user-facing failures.
- If something is uncertain, label it clearly.
- Exclude node_modules, build, dist, coverage, and .next unless relevant.
- Prefer issues backed by code paths, conditions, and data flow.
- Use tables where helpful.
- End with a prioritized action plan.

Output format:
1. Executive summary
2. Confirmed bugs
3. Highly likely bugs
4. Suspicious areas for manual QA
5. Recommended fixes in priority order

Please start by mapping the main user flows, then inspect the code to find bugs and produce the report.
