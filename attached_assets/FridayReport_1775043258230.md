## 1. New User Onboarding Journey

A three-part onboarding system was built to guide new users from sign-up to productive use of the platform.

### 1a. Guided Tour (`client/src/components/GuidedTour.tsx` — 272 lines)
- Step-by-step interactive tour that highlights key areas of the application using tooltip poppovers anchored to real UI elements.
- Covers: sidebar navigation, project list, task management, creating items, and AI features.
- Automatically triggers for new users who haven't completed the tour.
- Progress is persisted via the `/api/user-journey/status` and `/api/user-journey/checklist-progress` endpoints so it survives page reloads and sessions.

### 1b. First Project Wizard (`client/src/components/FirstProjectWizard.tsx` — 496 lines)
- Full-screen modal wizard that walks users through creating their very first project.
- Multi-step flow: project name/description, add tasks with optional start/end dates and priority, assign resources to tasks.
- Each task can be assigned to an existing team member (resource) with a resource selector dropdown.
- On completion, the wizard creates the project and all tasks via the existing REST API, then assigns resources using `PUT /api/tasks/:taskId/resources`.
- Wizard does not re-open after a project is created; progress state keys are validated.

### 1c. Getting Started Checklist (`client/src/components/GettingStartedChecklist.tsx` — 144 lines)
- Persistent checklist widget displayed on the Home page.
- Tracks milestones: create a project, add a task, assign a team member, use AI features, explore the dashboard.
- Progress auto-updates as users complete each action via the `use-user-journey` hook.
- Dismissable once all items are completed.

### 1d. User Journey Hook (`client/src/hooks/use-user-journey.ts` — 101 lines)
- Shared React hook managing onboarding state (tour completion, wizard completion, checklist progress).
- Communicates with `GET /api/user-journey/status` and `PUT /api/user-journey/checklist-progress` endpoints.
- Provides `trackChecklistEvent()` function used across the app (e.g., after AI create, after task creation).

### Backend Support
- **`server/routes/dashboardRoutes.ts`**: Added user journey status and progress persistence endpoints.
- **`shared/models/auth.ts`**: Extended user model to include journey state fields.
- **`client/src/pages/Home.tsx`**: Integrated the GettingStartedChecklist and GuidedTour components.
- **`client/src/components/layout/Sidebar.tsx`**: Tour target annotations added to sidebar links.

## 3. AI Project Agent (Major Feature)

A fully automated AI agent system that runs three scheduled actions per project: meeting agenda generation, task follow-up emails, and status reports. This is the largest feature in this change set.

### 3a. Database Schema (`shared/schema.ts`)

**`project_agents` table** — stores agent configuration per project:
| Column | Type | Purpose |
|--------|------|---------|
| `id` | serial PK | Unique agent ID |
| `projectId` | integer FK | Links to project |
| `organizationId` | integer FK | Organization scope |
| `enabled` | boolean | Master on/off toggle |
| `agendaEnabled` | boolean | Enable meeting agenda action |
| `agendaDay` | integer (0-6) | Day of week for agenda (default: Monday) |
| `agendaTime` | text | Time of day for agenda (default: "09:00") |
| `taskFollowUpEnabled` | boolean | Enable task follow-up action |
| `taskFollowUpDay` | integer (0-6) | Day of week (default: Wednesday) |
| `taskFollowUpTime` | text | Time of day (default: "09:00") |
| `statusReportEnabled` | boolean | Enable status report action |
| `statusReportDay` | integer (0-6) | Day of week (default: Friday) |
| `statusReportTime` | text | Time of day (default: "09:00") |
| `timezone` | text | IANA timezone (default: "America/New_York") |
| `lastAgendaRun` / `nextAgendaRun` | timestamp | Tracking fields |
| `lastTaskFollowUpRun` / `nextTaskFollowUpRun` | timestamp | Tracking fields |
| `lastStatusReportRun` / `nextStatusReportRun` | timestamp | Tracking fields |
| `createdBy` | varchar FK | User who enabled the agent |

Indexes: unique on `projectId`, indexed on `organizationId` and each `next*Run` timestamp for efficient cron queries.

**`project_agent_logs` table** — audit trail of every agent action:
| Column | Type | Purpose |
|--------|------|---------|
| `id` | serial PK | Log entry ID |
| `projectAgentId` | integer FK | Links to agent config |
| `projectId` | integer FK | Project context |
| `actionType` | text | "meeting_agenda", "task_follow_up", or "status_report" |
| `subject` | text | Email subject line |
| `recipientEmails` | text[] | Array of recipient emails |
| `emailPreview` | text | Truncated preview of email content (max 2000 chars) |
| `status` | text | "success", "error", or "skipped" |
| `errorMessage` | text | Error details if status is "error" |
| `createdAt` | timestamp | When the action was executed |

### 3b. Backend Service (`server/services/projectAgentService.ts` — 569 lines)

**Stakeholder Resolution (`getProjectStakeholders`)**
- Reads `managerId`, `businessSponsorId`, and `technicalLeadId` from the project record.
- Batch-resolves user IDs to email addresses via the `users` table.
- Returns a `StakeholderEmails` object with `managerEmail`, `sponsorEmail`, `techLeadEmail`.

**Email Template (`wrapEmailHtml`)**
- Generates a branded HTML email wrapper with a purple gradient header, content area, and footer with "Sent by AI Project Agent" branding.
- All emails include a link back to FridayReport.AI.

**Action 1: Meeting Agenda (`runMeetingAgenda`)**
- Queries the project's tasks and issues to build a context object including:
  - Recently completed tasks (up to 5)
  - In-progress tasks with progress percentages
  - Upcoming deadlines (within 7 days)
  - Overdue tasks
  - Open risks and issues
- Sends this context to OpenAI GPT-4o-mini with a system prompt requesting a structured agenda with sections: Wins & Accomplishments, Current Focus, Upcoming Deadlines, Blockers & Risks, Discussion Points, and Action Items.
- Emails the generated agenda to all configured stakeholders (PM, sponsor, tech lead).

**Action 2: Task Follow-Up (`runTaskFollowUp`)**
- Queries for two categories of tasks:
  - **Overdue**: tasks with `endDate < today` that are not completed/cancelled.
  - **Approaching**: tasks due within the next 3 days.
- For each task, resolves the assigned resource(s) and task owner to their email addresses.
- Groups tasks by recipient with deduplication (prevents the same task appearing twice for the same person).
- Sends personalized follow-up emails to each task owner listing their overdue and approaching tasks.
- Sends a comprehensive **PM Digest** to the project manager containing ALL incomplete tasks across the project (overdue, in-progress, not-started), grouped by category. This ensures the PM has a full picture, not just the items that triggered follow-ups.

**Action 3: Status Report (`runStatusReport`)**
- Generates a formatted HTML status report with:
  - Project health indicator (Green/Yellow/Red) with color-coded banner
  - Key metrics dashboard: completion %, total tasks, in-progress count, overdue count
  - High-priority risks and issues (Critical/High) highlighted in a red panel
  - Open items summary (risk count, issue count)
  - Direct link to the project in FridayReport.AI
- Emails the report to all stakeholders (PM, business sponsor, technical lead).

**Scheduling Engine (`calculateNextWeeklyRun`)**
- Calculates the next occurrence of a weekly scheduled run based on:
  - Target day of week (0=Sunday through 6=Saturday)
  - Target time of day (HH:MM format)
  - IANA timezone (handles DST transitions correctly via `date-fns-tz`)
- If the target time has already passed for this week, schedules for next week.

**Cron Runner (`checkAndRunDueAgentActions`)**
- Called every 15 minutes by the cron job registered in `server/index.ts`.
- Queries `project_agents` for agents with `enabled=true` and whose `next*Run` timestamp is in the past.
- Executes the due actions and updates `last*Run` / `next*Run` timestamps.
- Returns the count of actions executed (used for logging).

### 3c. REST API Routes (`server/routes/projectAgentRoutes.ts` — 279 lines)

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `GET` | `/api/projects/:projectId/agent` | Get agent configuration for a project |
| `GET` | `/api/projects/:projectId/agent/stakeholders` | Get resolved stakeholder info (names + emails) |
| `PUT` | `/api/projects/:projectId/agent` | Create or update agent configuration |
| `GET` | `/api/projects/:projectId/agent/logs` | Get agent activity log (limit 100, newest first) |
| `POST` | `/api/projects/:projectId/agent/trigger/:action` | Manually trigger an action (agenda, task_follow_up, status_report) |

All routes require authentication and verify the user has access to the project's organization. The trigger endpoint provides manual testing capability without waiting for the scheduled cron run.

## 5. Resource Assignment Bug Fix

### Root Cause
The `PUT /api/tasks/:taskId/resources` endpoint in `server/routes/resourceRoutes.ts` (line 728) was calling `recalculateTaskEstimatedHours(taskId)` — a function that was defined as a **local function inside `server/routes/projectItemRoutes.ts`** (line 763) but was never imported or defined in `resourceRoutes.ts`.

This caused a `ReferenceError` (HTTP 500) every time resources were assigned to tasks from:
- The Assign Resources dialog
- The First Project Wizard
- The AI chat create feature
- Any other caller of this endpoint

### Fix
Removed the broken `recalculateTaskEstimatedHours()` call from `resourceRoutes.ts`. The underlying storage function `updateTaskResourceAssignments()` in `server/storage/resourceStorage.ts` (lines 439-471) already handles estimated hours recalculation internally, making the removed call both broken and redundant.

**File changed:** `server/routes/resourceRoutes.ts` (7 lines removed)

## 6. AI Create "Assign to Me" Toggle Fix

### Problem
In the AI Create confirmation dialog (`client/src/components/layout/AICreateButton.tsx`), the "Assign all created tasks to you" toggle was treated as an independent action:
- It counted toward the total action count (e.g., "1 of 8 actions selected" when only the assignment was toggled on).
- Users could enable it without any task creation actions, leading to a no-op execution that appeared successful but did nothing.
- The "Create 1 Item" button would be enabled with only the assignment action selected.

### Fix
- **Toggle dependency**: When all task creation actions are toggled off, the assignment action automatically disables itself.
- **Count exclusion**: The assignment action is excluded from the action count display (only "creatable" actions like tasks, projects, risks, etc. are counted).
- **Switch disabled state**: The assignment toggle switch is visually grayed out and disabled when no task actions are enabled, providing clear feedback.

**File changed:** `client/src/components/layout/AICreateButton.tsx` (10 lines changed)

## Architecture Notes

- The cron job for the AI Project Agent runs every 15 minutes (registered in `server/index.ts` alongside existing cron jobs for scheduled reports and timesheet reminders).
- Agent configuration is 1:1 with projects (enforced by a unique index on `projectId`).
- All agent actions are fully idempotent — re-running them produces a new email but does not corrupt state.
- Agent logs are capped at 2,000 characters per entry to prevent database bloat from large HTML emails.
- The stakeholder lookup is shared between the agent service and the API routes, using the same `getProjectStakeholders` function.
