You are a senior enterprise architect, security engineer, and full-stack developer.

Your task is to implement a **Roles and Permissions system** in this already developed **PPM (Project Portfolio Management) solution**, using the existing architecture, data models, UI patterns, and business workflows already present in the codebase.

The goal is to add a robust authorization model so the application can control access to portfolio, project, resource, financial, timesheet, admin, and configuration features in a structured and scalable way.

## Main objective

Design and implement a complete authorization layer that supports:

* roles
* permissions
* role-based access control
* optional fine-grained permissions by feature/action
* user-role assignment
* access checks in backend and frontend
* visibility restrictions in menus, pages, actions, and APIs
* safe defaults and deny-by-default behavior where appropriate

The implementation must fit the current PPM solution, not an isolated demo module.

## What to do first

1. Inspect the existing codebase and identify:

   * auth/login/session mechanism
   * user model
   * organization, portfolio, project, team, and resource models
   * current admin/configuration modules
   * existing access restrictions, if any
   * API structure
   * frontend routing/navigation structure
   * database and ORM patterns
   * validation and middleware patterns

2. Determine what is already present:

   * existing roles
   * hardcoded admin checks
   * feature flags used as pseudo-permissions
   * tenant/org boundaries
   * project-level ownership concepts
   * manager/approver logic
   * timesheet approver logic
   * financial approval or governance logic

3. Then implement the authorization system in a way that reuses current conventions and minimizes disruption.

## Authorization model

Implement a flexible model that supports:

* **Role-based access control (RBAC)** as the main model
* optionally, **permission-based checks** for finer control
* support for both:

  * global/system roles
  * contextual roles where useful (for example project-level or portfolio-level roles)

Examples of roles that may be relevant in a PPM system:

* System Administrator
* PMO Administrator
* Portfolio Manager
* Project Manager
* Resource Manager
* Finance Manager
* Timesheet Approver
* Executive Viewer
* Team Member
* Read-Only User

Do not blindly create all of these if the current app structure does not support them. Adapt to what the repository actually contains.

## Business rules

Implement sensible enterprise rules such as:

* users should only see what they are authorized to view
* users should not be able to call restricted APIs even if the frontend hides the action
* menu items, buttons, pages, and form actions should respect permissions
* sensitive financial and admin data should be restricted
* timesheet approvers should only approve for allowed users/teams/projects if such concepts exist
* project managers may manage only their own projects if that matches the current domain model
* deny access by default when a permission is missing
* unauthorized actions must return proper HTTP/API errors
* inactive users should not retain active access
* role or permission changes should take effect safely and consistently
* seeded admin access should exist if the system needs an initial administrator

## Important rules

* Base everything on the actual codebase.
* Reuse existing patterns and naming conventions.
* Do not produce a generic RBAC essay detached from the repository.
* Clearly separate:

  * what already exists
  * what is missing
  * what you are adding
* Do not rely only on frontend hiding; enforce permissions in backend logic too.
* Be explicit about assumptions and uncertainties.
* Prefer maintainable and scalable design over quick hacks.
* If the app already has partial role logic, refactor and extend it instead of duplicating it.
* Keep the design enterprise-ready but not overengineered.

Start by inspecting the repository for authentication, user management, project/portfolio/resource/timesheet modules, existing admin logic, and any hardcoded access checks, then design and implement the roles and permissions system accordingly.
