You are a senior full-stack engineer and code auditor.

Your task is to inspect this application for **mismatches between the codebase and the schema/contracts it depends on**.

Focus on finding inconsistencies between:

* database schema and ORM models
* ORM models and API request/response types
* backend validation schemas and frontend form payloads
* API handlers and actual database fields
* TypeScript types/interfaces and real persisted data shape
* migrations and current code assumptions
* enum definitions across backend/frontend/database
* required vs optional fields
* field names, casing, nullability, defaults, and data types
* relationships, foreign keys, and cardinality assumptions
* schema docs / OpenAPI / GraphQL / Prisma / Zod / JSON Schema vs implementation

What to do:

1. Identify the authoritative schema sources in the repo.
2. Compare them against the application code.
3. Find all mismatches, broken assumptions, stale fields, renamed fields, missing migrations, invalid defaults, and incompatible types.
4. Flag places where the app may still compile but fail at runtime because the schema and code drifted.
5. Highlight the highest-risk issues first.

For each issue, provide:

* severity: critical / high / medium / low
* file(s) involved
* schema definition
* conflicting code usage
* why it is a mismatch
* likely runtime impact
* exact fix recommendation

Then produce these sections:

## Executive Summary

Short summary of the most important schema/code mismatches.

## Findings

A detailed list of every mismatch found.

## Likely Runtime Failures

Concrete examples of requests, queries, or UI flows that may break.

## Recommended Fix Order

Prioritized remediation plan.

## Patch Suggestions

Show precise code changes or migration updates needed.

Important rules:

* Do not give generic advice unless tied to an actual mismatch you found.
* Be strict and evidence-based.
* Quote exact field names and types.
* Call out uncertainty explicitly when the repository does not provide enough proof.
* If no schema is found, say that clearly and infer likely contracts from usage.
* Prefer concrete findings over broad commentary.

If possible, also generate a compact table like:

| Severity | Area | Schema | Code | Problem | Fix |

Start by listing which schema sources you found and which parts of the app you compared them against.
