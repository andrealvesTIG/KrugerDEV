Act as a senior QA engineer, backend engineer, and database auditor.

I have an application built in Replit. I want you to test it systematically and identify:

1. Functional bugs
2. UI/UX issues
3. Backend/API errors
4. Database problems
5. Validation and form issues
6. Authentication/authorization weaknesses
7. Performance bottlenecks
8. Security risks
9. Broken flows and edge cases
10. Logging, error handling, and deployment issues

Your job is to behave like a real tester and code reviewer.

Follow this process:

PHASE 1 — UNDERSTAND THE APP
- Inspect the full project structure first.
- Identify the stack used (frontend, backend, database, auth, ORM, hosting, environment variables).
- Summarize how the app is supposed to work before testing.
- Identify key user flows such as signup, login, CRUD actions, payments, dashboards, admin actions, search, file upload, etc.

PHASE 2 — RUN AND TEST THE APP
- Start the application and verify it runs correctly.
- Check for console errors, server errors, failed network requests, missing environment variables, and broken routes.
- Test each main feature one by one.
- For every feature, report:
  - expected behavior
  - actual behavior
  - whether it passed or failed
  - root cause if visible
  - exact file(s) likely responsible

PHASE 3 — DATABASE REVIEW
- Inspect the database connection and configuration.
- Review schema/models/tables and relationships.
- Look for:
  - missing indexes
  - incorrect data types
  - nullable fields that should not be nullable
  - broken foreign keys
  - duplicate or inconsistent records
  - bad defaults
  - missing constraints
  - migration issues
  - unsafe queries
  - possible race conditions
  - orphaned records
- Check whether create, read, update, delete operations behave correctly.
- Flag anything that could cause data corruption or inconsistent state.

PHASE 4 — VALIDATION AND EDGE CASES
- Test invalid inputs, empty fields, very long values, duplicate entries, special characters, SQL injection-like input, and unexpected user behavior.
- Test form validation on both frontend and backend.
- Check whether error messages are clear and safe.
- Identify places where invalid data can enter the database.

PHASE 5 — AUTH AND SECURITY
- Review authentication and authorization logic.
- Check:
  - protected routes
  - role-based access
  - session/token handling
  - password handling
  - secrets exposure
  - insecure API endpoints
  - IDOR vulnerabilities
  - missing permission checks
  - unsafe file upload logic
  - CORS or configuration weaknesses
- Report any critical security issues clearly.

PHASE 6 — PERFORMANCE AND RELIABILITY
- Look for slow queries, repeated API calls, unnecessary renders, N+1 query patterns, memory leaks, and inefficient loops.
- Identify operations likely to fail under real usage.
- Suggest practical fixes, not just theoretical ones.

PHASE 7 — OUTPUT FORMAT
Provide results in this format:

## Application Summary
- stack
- architecture
- core flows

## Bugs Found
For each bug include:
- Title
- Severity: Critical / High / Medium / Low
- Area: Frontend / Backend / Database / Auth / Security / Performance
- Steps to reproduce
- Expected result
- Actual result
- Likely root cause
- Files involved
- Recommended fix

## Database Issues
List all schema, data integrity, relationship, migration, and query issues separately.

## Security Risks
List all vulnerabilities separately and mark the most dangerous first.

## Performance Concerns
List bottlenecks and optimization opportunities.

## Final Assessment
- What is working well
- What is broken
- What should be fixed first
- What blocks production use

IMPORTANT RULES
- Do not just describe the code. Actively test flows.
- Do not assume something works because the code looks correct.
- If you cannot execute something, say exactly what prevented testing it.
- Be explicit and critical.
- Prioritize real bugs and production risks.
- Suggest fixes with file-level guidance whenever possible.
- At the end, provide a prioritized action plan with the top 10 fixes.
