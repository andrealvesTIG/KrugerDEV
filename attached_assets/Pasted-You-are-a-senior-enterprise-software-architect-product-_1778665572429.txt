You are a senior enterprise software architect, product manager, and full-stack engineer.

Your task is to design and implement an **Enterprise Calendar capability** in this application, following the **same business logic and operational behavior as Microsoft Project Online enterprise calendars**, adapted to the architecture and patterns already used in this Replit application.

The goal is to create a calendar system that supports **working time rules, exceptions, holidays, resource calendars, and scheduling logic** so the tool can calculate dates, durations, allocations, and non-working time consistently across the application.

## Main objective

Add an enterprise-grade calendar module that allows the application to:

* define organization-wide working calendars
* configure standard working days and working hours
* configure weekends and non-working days
* define holidays and date exceptions
* define recurring exceptions
* support resource-specific or team-specific calendars
* calculate start/end dates based on working time
* skip non-working time when scheduling tasks
* support partial working days
* support inherited calendars or base calendars
* behave similarly to Microsoft Project Online calendar logic where relevant

## Functional requirements

Implement support for the following concepts:

### 1. Base / Enterprise Calendars

The system must support one or more enterprise calendars, such as:

* Standard
* Night Shift
* Regional Calendar
* Team Calendar
* Plant / Operations Calendar

Each calendar should support:

* name
* description
* status (active/inactive)
* timezone if applicable
* default working week definition
* exceptions and holiday overrides

### 2. Working Week Logic

Each enterprise calendar should allow configuration of:

* working days of the week
* multiple working intervals per day
* non-working days
* half-days
* lunch-break style gaps between working periods

Example:

* Monday to Friday:

  * 08:00–12:00
  * 13:00–17:00
* Saturday and Sunday:

  * non-working

### 3. Calendar Exceptions

Support date-specific exceptions such as:

* public holidays
* company shutdown periods
* special overtime days
* reduced working hours
* emergency maintenance days
* one-time non-working dates
* one-time working overrides on otherwise non-working days

Each exception should support:

* name
* start date
* end date
* all-day or time-bound
* working / non-working override
* optional custom working intervals

### 4. Recurring Exceptions

Support recurring holiday or rule definitions like:

* every January 1st
* first Monday of September
* every December 25th
* annual shutdown periods
* monthly exceptions if needed

The implementation should clearly distinguish:

* one-time exceptions
* recurring exceptions
* generated occurrence handling if needed

### 5. Resource and Team Calendars

Allow the system to assign calendars to:

* organization
* project
* department/team
* individual resource/user

The scheduling precedence should be clearly defined, for example:

1. resource calendar
2. team/project calendar
3. enterprise default calendar

If the current application already has users, teams, projects, or assignments, integrate with those models instead of inventing disconnected structures.

### 6. Scheduling Logic

Implement or propose logic so the application can:

* calculate working duration between two timestamps
* calculate finish date from start date + effort/duration
* calculate start date from finish date - effort/duration
* move dates to next valid working slot
* detect whether a given datetime is working or non-working
* skip weekends, holidays, and exception periods
* support tasks spanning multiple days with split working windows

The behavior should resemble Microsoft Project Online scheduling logic as closely as practical.

### 7. Calendar Administration UI

Add or propose UI screens/components for:

* list of enterprise calendars
* create/edit calendar
* weekly working time editor
* exception manager
* holiday manager
* calendar assignment to resources/teams/projects
* preview / simulation mode for date calculations
* audit/history if the app already supports change tracking

### 8. API / Backend Support

Create or update:

* database schema
* ORM models
* service layer
* validation schemas
* API endpoints
* permissions / role checks
* tests

The solution must fit the current app structure and coding conventions.

## Business rules

Follow these scheduling rules unless the existing system already has stronger constraints:

* non-working time must never count toward task duration
* a task starting outside working time should move to the next valid working slot
* a task ending calculation must stop at the end of working intervals and continue on the next valid interval
* holidays and non-working exceptions override standard working week rules
* working exceptions can override otherwise non-working days
* resource-specific calendars should override broader calendars
* inactive calendars should not be assignable
* deleting calendars in use should be prevented or soft-deleted safely
* calendar edits should not silently corrupt historical scheduling data
