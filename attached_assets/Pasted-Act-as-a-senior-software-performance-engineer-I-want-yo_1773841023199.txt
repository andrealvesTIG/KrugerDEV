Act as a senior software performance engineer.

I want you to inspect this codebase and identify files that are unusually large and could hurt application performance, maintainability, build times, or developer experience.

Your task:

1. Scan the entire project and list the files that are “too long” or unusually heavy.
2. For each flagged file, include:
   - file path
   - total number of lines
   - file type
   - why it may be a problem
   - likely impact:
     - runtime performance
     - bundle size
     - build/compile time
     - hot reload/dev server speed
     - readability/maintainability
3. Prioritize files that are especially important if they are:
   - frontend pages/components
   - backend handlers/services
   - utility files doing too many things
   - huge config files
   - generated files accidentally committed
   - files with large embedded JSON, mock data, SVG, SQL, HTML, or constants
4. Detect common anti-patterns in long files, such as:
   - multiple responsibilities in one file
   - very large React/Vue components
   - giant switch/if blocks
   - large static datasets in source code
   - duplicated code
   - deeply nested logic
   - large inline styles/templates
   - large helper files used everywhere
5. Give me a ranked report with these sections:
   - Critical: files that should be split immediately
   - Warning: files that are getting too large
   - Acceptable but worth watching
6. Suggest concrete refactors for each critical file, including:
   - how to split it
   - which parts should move to separate modules
   - whether any content should be lazy-loaded, virtualized, cached, paginated, or moved out of source code
7. If possible, also identify:
   - very large imports
   - files likely increasing bundle size
   - files with large in-memory objects/constants
   - routes/pages loading too much at startup
8. At the end, provide:
   - Top 10 longest files
   - Top 10 files most likely to affect performance
   - quick wins I can do today
   - a recommended max file length by file type

Rules:
- Exclude obvious dependency/vendor/build folders unless something unusual is inside them:
  - node_modules
  - .git
  - dist
  - build
  - coverage
  - .next
  - out
  - vendor
- Ignore minified/generated files unless they are committed in a suspicious place and should be called out.
- Be practical, not dogmatic: a long file is only a problem if it likely affects performance, complexity, or maintenance.
- If you cannot measure actual runtime performance, clearly separate:
  - “likely performance risk”
  - “maintainability risk”
- Use a table for the main findings.
- End with a concise action plan in priority order.

Scoring guidance:
- < 200 lines: usually fine
- 200–400 lines: review if complex
- 400–800 lines: warning
- 800+ lines: likely needs refactor
- 1500+ lines: critical unless clearly justified

Please start by scanning the repo and then produce the full report.
