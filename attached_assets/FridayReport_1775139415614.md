### Add detailed organization activity breakdown to the dashboard
**Commit:** `dbcdff26`

Introduces a new "Organizations" tab to the User Activity KPI Dashboard, providing a per-organization breakdown of various activities with time period filtering and detailed metrics.

**Technical Implementation:**

- **New API endpoint:** `GET /api/admin/user-activity-kpi/organizations`
  - Protected with `requireSuperAdmin()` — only super admin users can access.
  - Accepts a `period` query param (`7d`, `14d`, `30d`, `90d`, `6m`, `1y`, `all`) which maps to a PostgreSQL `INTERVAL`. Default is `30d`.
  - Dynamic SQL filter construction using Drizzle ORM's `sql` template literal. A helper `dateFilterAlias(alias)` generates aliased date filters for joined queries (e.g., `AND t.created_at >= NOW() - INTERVAL '30 days'`).
  - Iterates over all active organizations (`WHERE deactivated_at IS NULL`) and runs individual queries per org for: projects, tasks, risks, issues, timesheet entries/hours, reports, imports/exports, integrations, resources, portfolios, and activity logs.
  - Computes `totals` object by reducing across all organization metrics on the server side.
  - Returns `{ organizations, totals, period }`.

- **Frontend** (`client/src/components/admin/UserActivityKPITab.tsx`):
  - New `OrganizationsView` component added as a fifth tab (`ViewMode = "organizations"`).
  - Fetches data via React Query with key `["/api/admin/user-activity-kpi/organizations", period]`.
  - Summary cards row using `motion.div` animations showing: total orgs, projects, tasks, risks, issues, and timesheet entries.
  - Client-side search filtering by org name (case-insensitive `.includes()`).
  - Client-side sorting with options: most projects, most tasks, most members, most timesheets, name A-Z.
  - Per-org expandable cards showing all metric categories with `Progress` bar components indicating relative contribution.
  - Uses icons from `lucide-react`: `Building2`, `FolderKanban`, `ListTodo`, `AlertTriangle`, `Bug`, `Clock`, `FileText`, `Upload`, `Plug`, `Briefcase`.
  - Period selector `<Select>` component with options synced to the API's accepted values.

**Things to Avoid:**
- The API iterates organizations and runs ~12 individual queries per org in a `for` loop. This is an N+1 query pattern — if the number of organizations grows large, this will degrade performance. Consider refactoring to batch queries using `GROUP BY organization_id` across a single query per metric type.
- The `dateFilterAlias` helper uses `sql.raw(alias)` for table alias injection. Do not pass user-controlled strings into this — it would create a SQL injection vector. Currently safe because the alias is hardcoded.
- Client-side search and sorting is fine for small org counts but won't scale. If org count exceeds ~100, move filtering/sorting to the server.
- The `period` switch/case defaults to `30d` for unrecognized values — this is silent. Consider returning a 400 error for invalid period values to make debugging easier.

## March 31, 2026

### Fix error when loading user activity data
**Commit:** `35467e9c`

Corrected a SQL query issue in the weekly retention calculation to properly handle correlated subqueries.

**Technical Implementation:**

- **The Bug:** The original weekly retention query used a correlated subquery inside `SELECT` that referenced `ual.created_at` from the outer `GROUP BY` clause:
  ```sql
  SELECT DATE_TRUNC('week', ual.created_at) as week_start,
    COUNT(DISTINCT ual.user_id) as active_users,
    (SELECT COUNT(*) FROM users WHERE created_at <= DATE_TRUNC('week', ual.created_at) + INTERVAL '7 days') as total_users
  FROM user_activity_logs ual
  WHERE ual.created_at >= NOW() - INTERVAL '12 weeks'
  GROUP BY DATE_TRUNC('week', ual.created_at) ORDER BY week_start
  ```
  PostgreSQL rejected this because the correlated subquery referenced a non-grouped column (`ual.created_at`) through `DATE_TRUNC` in the `SELECT` list while also having a `GROUP BY`.

- **The Fix:** Wrapped the outer query as a subquery aliased `w`, then ran the correlated subquery against `w.week_start`:
  ```sql
  SELECT w.week_start, w.active_users,
    (SELECT COUNT(*) FROM users WHERE created_at <= w.week_start + INTERVAL '7 days') as total_users
  FROM (
    SELECT DATE_TRUNC('week', ual.created_at) as week_start,
      COUNT(DISTINCT ual.user_id) as active_users
    FROM user_activity_logs ual WHERE ual.created_at >= NOW() - INTERVAL '12 weeks'
    GROUP BY DATE_TRUNC('week', ual.created_at)
  ) w ORDER BY w.week_start
  ```

**Things to Avoid:**
- Never reference raw columns from the outer query inside a correlated subquery when the outer query has a `GROUP BY`. PostgreSQL requires the column to either be in the `GROUP BY` or be wrapped in an aggregate — or, as done here, materialize the grouped result first.
- The correlated subquery (`SELECT COUNT(*) FROM users WHERE ...`) still runs once per week. Over a 12-week window this is manageable, but if extended, consider using a window function or CTE instead.

### Add platform-wide user activity analytics for super administrators
**Commit:** `fd66c561`

Migrated the user activity KPI dashboard from the organization-level dashboard to the super admin area. Updated the API endpoint from `/api/dashboard/user-activity-kpi` to `/api/admin/user-activity-kpi` and removed organization-scoped filtering so the data reflects platform-wide activity.

**Technical Implementation:**

- **API endpoint:** `GET /api/admin/user-activity-kpi` in `server/routes/miscRoutes.ts`
  - Auth: `requireSuperAdmin(userId)` — replaces the previous `userHasOrgAccess()` check.
  - Removed all `organization_members` JOIN filtering — queries now scan the entire `users` and `user_activity_logs` tables.
  - KPI metrics computed:
    - `totalUsers`: `COUNT(*) FROM users`

    - `avgActionsPerUser`: Subquery computing per-user action count, then `AVG()` over the result.
    - `overallRetentionRate`: Users who existed 7+ days ago and were active in the last 7 days, divided by all users created 7+ days ago.
    - `newUsersThisWeek` / `newUsersLastWeek`: Simple `COUNT` with interval-based `WHERE` on `users.created_at`.
  - **Cohort analysis:** Defines `periodDefs` array with lifecycle stages (Week 1 through 12 months). For each period, queries `user_activity_logs` joined with `users` to compute active count, total actions, avg actions per user, and top 5 actions.
  - **Action breakdown:** Top 20 actions from the last 90 days with percentage calculation. Action names are humanized using `String.replace(/_/g, ' ').replace(/\b\w/g, c => c.toUpperCase())`.
  - **Daily active users:** 30-day window, grouped by date with timezone conversion: `(created_at AT TIME ZONE 'UTC') AT TIME ZONE 'America/New_York'`.
  - **Weekly retention trend:** 12-week rolling window (fixed in commit `35467e9c`).

- **Frontend** (`client/src/components/admin/UserActivityKPITab.tsx`):
  - Moved from `client/src/components/dashboard/UserActivityKPIDashboard.tsx`.
  - Removed `useOrganization()` hook and `organizationId` query param.
  - Query key changed from `["/api/dashboard/user-activity-kpi", currentOrganization?.id]` to `["/api/admin/user-activity-kpi"]`.
  - 4 view modes: Overview (area charts, pie charts), Cohorts (stacked bar + radar charts), Retention (heatmap + line), Engagement (composed charts).
  - Uses `recharts` library: `AreaChart`, `BarChart`, `LineChart`, `PieChart`, `RadarChart`, `ComposedChart` with `ResponsiveContainer`.
  - `framer-motion` animations with `fadeIn` variants (`opacity: 0, y: 20` → `opacity: 1, y: 0`).
  - KPI cards use color-coded borders: green (>=50%), amber (>=25%), red (<25%) for retention.
  - Cohort colors defined as a 12-element array of hex values.

- **Removed files:**
  - `server/routes/dashboardRoutes.ts` — old org-scoped endpoint (268 lines removed).
  - `client/src/components/dashboard/index.ts` — old export.
  - `client/src/pages/Dashboard.tsx` — removed tab integration.

- **Added integration:**
  - `client/src/pages/SuperAdmin.tsx` — imported and rendered `UserActivityKPIDashboard` as a tab.

**Things to Avoid:**
- The daily active users query hardcodes `'America/New_York'` timezone. If users are in different timezones, DAU counts will be skewed. Consider making the timezone configurable or using UTC consistently.
- Action name humanization (`replace(/_/g, ' ')`) is done on the server response. If action names contain multi-word segments or special characters, the output can be misleading (e.g., `view_project_details` becomes `View Project Details` but `api_v2_call` becomes `Api V2 Call`).
- The old endpoint `/api/dashboard/user-activity-kpi` was removed entirely. Any client code or bookmarks referencing it will break with no redirect — this is intentional since it was org-scoped.
- All queries run as separate `db.execute(sql`...`)` calls. There's no transaction wrapping, so data could be slightly inconsistent between metrics if activity happens during the request. Acceptable for analytics but worth noting.

  - Uses `useMemo` for derived data: `retentionHeatmapData`, `engagementByPeriod`, `weekOverWeekGrowth`.
  - Chart rendering with `recharts`:
    - Overview: `AreaChart` (DAU), `BarChart` (retention by lifecycle), `PieChart` (action breakdown), `ComposedChart` (engagement).
    - Cohorts: Cohort selector dropdown, `BarChart` per cohort, `RadarChart` for top actions.

**Things to Avoid:**
- This version was org-scoped and was replaced in the very next commit (`fd66c561`). Do not reference the `/api/dashboard/user-activity-kpi` endpoint — it no longer exists.
- The `LEFT JOIN organization_members` approach for org filtering means users who belong to multiple organizations will be counted in each org's metrics. If you re-introduce org-level filtering, consider whether this is the desired behavior.
- The `orgFilter` SQL fragment is appended directly with `${orgFilter}`. Be careful not to accidentally omit the `AND` keyword when constructing dynamic filters this way — it will cause a syntax error that's hard to debug because the SQL is built from template literals.

### Add Marketing section to SuperAdmin Analytics for selfie leads
**Commits:** `23b2375a`, `d2f6d6cc`, `941ff248`, `0efd117f` (iterated through code review)

Added a new "Marketing" sub-tab to the SuperAdmin Analytics/Monitoring section displaying selfie leads captured from the UnCon 2026 selfie page.

**Technical Implementation:**

- **API endpoint:** `GET /api/admin/selfie-leads` in `server/routes/userRoutes.ts`
  - Auth: Strict `super_admin` role check (not `hasAdminAccess` which includes org admins).
  - Query params:
    - `page` (default: 1), `limit` (default: 50, max: 100 for normal reads).
    - `search` — server-side ILIKE on `name`, `email`, `interviewer` columns: `WHERE name ILIKE '%term%' OR email ILIKE '%term%' OR interviewer ILIKE '%term%'`.
    - `export=true` — bypasses the 100-per-page cap (sets limit to 999999) for Excel export. This is the only way to fetch the full dataset.
  - Pagination offset: `(page - 1) * limit`.
  - Returns `{ leads, pagination: { page, limit, total, totalPages } }`.
  - Summary stats (total leads, unique emails, interviewer count) are always computed from the full unfiltered dataset, regardless of search/page params.
  - Data source: `uncon_selfie_leads` table via Drizzle `unconSelfieLeads` schema, imported dynamically with `await import("@shared/schema")`.

- **Frontend** (`client/src/components/admin/MarketingTab.tsx`):
  - React Query: `queryKey: ["/api/admin/selfie-leads", page, debouncedSearch]`.
  - Search debounce: 300ms delay using `setTimeout`/`clearTimeout` pattern. Resets to page 1 on search change.
  - Client-side column sorting: `sortField` + `sortDirection` state. Sorts the current page data only (not the full dataset). Sortable columns: name, email, interviewer, date.
  - Photo thumbnails: Rendered as `<img>` with `object-cover`, `cursor-pointer`. On click, opens a full-size preview in a `Dialog` component.
  - Excel export: Triggered by button click. Fetches all data via `export=true` query param, then generates an `.xlsx` file using the `xlsx` library (`XLSX.utils.json_to_sheet()` → `XLSX.writeFile()`). Only exports Name, Email, Interviewer, and Date/Time columns. Shows a toast notification on failure.
  - Pagination: Prev/Next buttons with page number display. Button disabled states based on `page === 1` and `page === totalPages`.
  - Summary cards: Total Selfie Leads (from `pagination.total`), Unique Emails (client-side `new Set(leads.map(l => l.email)).size`), Interviewers (client-side unique count).

- **Integration** (`client/src/components/admin/MonitoringTab.tsx`):
  - Added `'marketing'` to the `MonitoringSubTab` union type.
  - Added a button with `Megaphone` icon from `lucide-react` to the sub-tab navigation bar.
  - Refresh handler: `queryClient.invalidateQueries({ queryKey: ["/api/admin/selfie-leads"] })` when on the marketing tab.

**Things to Avoid:**
- The `export=true` flag sets the limit to 999999 — this fetches the entire dataset in one query. If the leads table grows to tens of thousands of rows, this will cause memory pressure and slow responses. Consider streaming the export or implementing server-side Excel generation.
- Unique email and interviewer counts in the summary cards are computed client-side from the current page data, but the total count comes from the server's full dataset. This creates an inconsistency — the summary card says "Total: 500" but the unique counts only reflect what's on the current page. The final iteration fixed global summary stats to come from the server.
- The dynamic schema import (`await import("@shared/schema")`) inside the route handler is unusual. This was likely done to avoid circular imports. If this pattern is reused, ensure the import is cached by Node.js module resolution and doesn't re-execute on every request.
- Client-side column sorting only sorts within the current page, not across the full dataset. Users may find this confusing. Either document this behavior or implement server-side sorting.

### Comprehensive Database & Backend QA Audit
**Commits:** `c68f229a`, `3aa3939c`, `f704416d`

Performed a thorough audit and cleanup of database storage patterns, replacing manual "check-then-insert/update" patterns with atomic upsert operations.

**Technical Implementation:**

- **`server/storage/miscStorage.ts` — Custom field values and project scores:**
  - `upsertProjectCustomFieldValue()`: Replaced a manual `getProjectCustomFieldValue()` → conditional `update()`/`insert()` with a single `db.insert().onConflictDoUpdate()` using Drizzle ORM.
    - Conflict target: `[projectCustomFieldValues.projectId, projectCustomFieldValues.fieldDefinitionId]` (composite unique index).
    - On conflict: Updates `value` and `updatedAt`.
  - `upsertProjectScore()`: Same refactor — replaced `db.select()` → conditional branch with `db.insert().onConflictDoUpdate()`.
    - Conflict target: `[projectScores.projectId, projectScores.criteriaId]`.
    - On conflict: Updates `score`, `justification`, `scoredBy`, `scoredAt`.

- **`server/storage/organizationStorage.ts` — External shares:**
  - `createExternalShare()`: Replaced a multi-step flow (check existing → check if revoked → update or return existing → insert new) with a single `db.insert().onConflictDoUpdate()`.
    - Conflict target: `[externalShares.objectType, externalShares.objectId, externalShares.sharedWithUserId]`.
    - On conflict: Sets `revokedAt: null`, updates `accessRole` and `sharedBy` (effectively "re-shares" a previously revoked share).

- **`server/storage/timesheetStorage.ts` — Timesheet settings:**
  - `upsertTimesheetSettings()`: Replaced check-then-insert/update with `db.insert().onConflictDoUpdate()`.
    - Conflict target: `[timesheetSettings.organizationId]`.
    - On conflict: Spreads the settings object and updates `updatedAt`.

- **`shared/schema.ts` — Added missing unique index:**
  - Added `uniqueIndex("external_shares_obj_user_idx").on(table.objectType, table.objectId, table.sharedWithUserId)` to the `externalShares` table definition. This is required for the `onConflictDoUpdate` to work — without a unique constraint, PostgreSQL cannot identify the conflict target.

**Things to Avoid:**
- When using `onConflictDoUpdate`, ensure the conflict target columns have a unique index or constraint defined in the schema. Without it, Drizzle will generate invalid SQL and the query will fail at runtime with a PostgreSQL error about missing unique constraint.
- The `createExternalShare()` refactor changed behavior: the old code would return the existing (non-revoked) share without modifying it. The new code will always overwrite `accessRole` and `sharedBy` even if the share already exists and is active. This means re-sharing with different permissions will silently update the existing share rather than being a no-op. Verify this is the intended behavior.
- The `upsertTimesheetSettings()` uses object spread (`{ ...settings, updatedAt: new Date() }`) in the `set` clause. If `settings` includes fields like `id` or `organizationId`, these will be included in the `SET` clause. While PostgreSQL will accept setting a column to its own value, it's wasteful. Consider explicitly listing only the mutable fields.
- When adding unique indexes to existing tables (like `external_shares_obj_user_idx`), ensure a database migration is run. If duplicate rows already exist that violate the new constraint, the migration will fail. Run a deduplication query first: `DELETE FROM external_shares WHERE id NOT IN (SELECT MIN(id) FROM external_shares GROUP BY object_type, object_id, shared_with_user_id)`.

### PMO Radar: Add realistic simulation with closures, escalations, and new synthetic items
**Commit:** `1281f94e`

Added a full simulation engine to the PMO Radar feature, enabling scenario-based risk/issue projections with realistic lifecycle transitions.

**Technical Implementation:**

- **Simulation Engine** (`client/src/lib/simulationEngine.ts`, 526 lines):
  - `seededRandom(seed)`: Deterministic pseudo-random number generator using multiplicative hashing (`seed * 2654435761`). Ensures simulation results are reproducible for the same inputs.
  - `runSimulation(signals, monthsAhead, generateNewItems, scenarioId)`: Main entry point.
    - Iterates over existing signals month-by-month, applying status transitions:
      - `Open` → `In Progress` (probability-based), `Open/In Progress` → `Mitigated`/`Closed` (based on risk score thresholds).
      - Escalation: Increases risk score by 1-2 points if random check passes.
    - Returns `SimulationResult` with `signals`, `closedCount`, `newCount`, `escalatedCount`, and `log` (array of `SimulationLogEntry`).
  - `generateSyntheticSignal(patterns, month, seed)`: Creates new risk/issue items during simulation.
    - Derives from `DatasetPatterns` (averages computed from active-only signals).
    - Title generation from `SYNTHETIC_TITLE_PREFIXES` keyed by category (schedule, budget, dependency, resource, technical, scope).
    - Cost exposure sampled around the dataset's median with random variance.
    - Synthetic signals get `isSimulated: true` flag.
  - `analyzeDatasetPatterns(signals)`: Computes averages from active (non-Closed, non-Mitigated) signals only. Extracts category distribution, project IDs/names/portfolios, risk-to-issue ratio.
  - `runExposureSimulation(signals, monthsAhead, generateNewItems)`: Time-series exposure calculation including new items, returned as `{ month, existingExposure, newExposure }[]`.

- **Simulation Scenarios** (`client/src/lib/simulationScenarios.ts`, 77 lines):
  - Predefined scenarios with configurable parameters (closure rate multiplier, escalation multiplier, new item rate).

- **Simulation Report Dialog** (`client/src/components/radar/SimulationReportDialog.tsx`, 371 lines):
  - Modal displaying simulation results: summary stats, log table with action icons, and charts.
  - Filterable by action type.

- **Filters Panel Updates** (`client/src/components/radar/FiltersPanel.tsx`, +320 lines):
  - "Generate New Items" toggle visible only when projection mode is active.
  - Simulation summary bar showing closed/mitigated, new, and escalated counts.
  - Legend entry for simulated items (dashed outline indicator).

- **Radar Canvas Updates** (`client/src/components/radar/RadarCanvas.tsx`):
  - `isSimulated` field added to `RiskSignal` type.
  - Simulated items rendered with dashed stroke (`strokeDasharray`) and reduced fill opacity (50%).

- **PMO Radar Page Updates** (`client/src/pages/PmoRadar.tsx`):
  - Replaced old hash-based projection with `runSimulation()` engine.
  - `generateNewItems` boolean state for toggling synthetic generation.
  - Exposure chart uses `baseFilteredSignals` (respects UI filters).
  - New item cost exposure shown as separate purple `<Area>` on the chart.

**Things to Avoid:**
- The `seededRandom` function is a simple hash — it has limited entropy and can produce visible patterns with sequential seeds. For more robust simulations, consider a proper PRNG like a linear congruential generator or xorshift.
- Synthetic signals are generated entirely on the client side and are never persisted to the database. If a user expects to save or share simulation results, this won't work. The `isSimulated: true` flag prevents editing but doesn't prevent confusion.
- The simulation runs synchronously on the main thread. With large signal datasets (1000+) and long projection periods (12+ months), this could cause UI jank. Consider moving to a Web Worker if performance becomes an issue.
- The edit button is hidden for simulated items in `DetailsDrawer`, but there's no server-side guard. If someone crafts an API request with a simulated item's ID, the server would return a 404 (since the item doesn't exist in the DB) — this is fine, but error handling should account for it gracefully.
- `analyzeDatasetPatterns` filters to active signals only. If all signals are closed/mitigated, the function will have `n = 1` (fallback) and produce default values, which may generate unrealistic synthetic signals.

### Partner Program page and navigation
**Commits:** `c8ab9de2`, `6cfe0555`, `2d5240a5`, `260bf219`, `96fcffc7`, `9a8626ca`, `b6e84374`, `84e8376d`, `8899fff1`

Created a full Partner Program landing page with application form, email notifications, and cross-site navigation.

**Technical Implementation:**

- **Database schema** (`shared/schema.ts`):

    - `name`: varchar, required.
    - `email`: varchar, required.
    - `company`: varchar, nullable.
    - `partnerType`: varchar, required — one of `"consulting"`, `"independent"`, `"trainer"`.
    - `message`: text, nullable.

- **API endpoint:** `POST /api/partner-applications` in `server/routes/partnerRoutes.ts`
  - **Validation:** Zod schema (`partnerApplicationRequestSchema`) validates all fields including honeypot fields and `formLoadTime` (number).
  - **Bot protection:**
    - Honeypot: Two hidden fields (`honeypot1`, `honeypot2`). If either is filled, the request silently returns `200 { success: true }` (bot doesn't know it failed).
    - Timing check: `Date.now() - formLoadTime`. If elapsed time is <500ms, the request is silently accepted without saving (humans can't fill a form that fast).
  - **Rate limiting:** IP-based, in-memory `Map<string, number[]>` tracking submission timestamps per IP. Window: 1 hour, max: 5 submissions per IP. Returns `429 Too Many Requests` when exceeded.
  - **Email notifications** (added in commit `260bf219`):
    - Uses `sendEmail()` from `server/services/email.ts` (Resend email service).

    - Both emails are fire-and-forget: `sendEmail().catch(err => console.error(...))` — they never block the API response or prevent the DB insert.
    - Partner type labels mapped via `PARTNER_TYPE_LABELS` constant: `{ consulting: "PMO Consulting Firm", independent: "Independent Consultant", trainer: "Trainer / Educator" }`.

- **Frontend** (`client/src/pages/PartnerProgramPage.tsx`, 546 lines):
  - Route: `/partners`, lazy-loaded via `React.lazy()` in `App.tsx`.
  - Uses `framer-motion` for scroll-triggered animations: `useInView(ref, { once: true, margin: "-60px" })`.
  - `AnimatedSection` wrapper component applying `staggerContainer` variants (0.12s stagger between children).
  - Partner types defined as a constant array with `id`, `title`, `icon`, `benefits` fields.
  - Form state managed via `useState` with `handleSubmit` using `apiRequest("POST", "/api/partner-applications", ...)`.
  - `HoneypotField` component provides hidden fields + `formLoadTime` tracking, passed up via `onHoneypotChange` callback.
  - Success state: replaces the form with a confirmation message after submission.
  - Light theme enforcement: `useForceLightTheme()` hook that removes `"dark"` class and adds `"light"` class to `<html>` on mount, restoring the previous theme on unmount.

- **Navigation integration:**
  - Added "Partners" link to 4 pages: `LandingPageNew`, `IndustryLandingPage`, `UnCon2026LandingPage`, and `SignInPage`.
  - Desktop nav: visible on `sm+` screens, styled consistently with existing nav (slate-200 text, orange hover).
  - Mobile mega-menu: added to the Navigate section.

- **Additional iterations:**
  - `96fcffc7`: Added official organization logos (PMI, PMO GA, IPMA, APM, AIPM) as image assets.
  - `9a8626ca`: Added consistent footer to partner page.
  - `b6e84374`: Removed unused organization logo imports.
  - `84e8376d`: Made partner org logos clickable links.
  - `8899fff1`: Added custom apply button labels per partner type card.

**Things to Avoid:**
- The rate limiter uses an in-memory `Map` — it resets on server restart and doesn't work across multiple server instances. For production, use Redis or a database-backed rate limiter.
- The honeypot returns `200 { success: true }` to bots. This means the frontend `handleSubmit` will show the success state even for bot submissions. This is intentional (to fool bots) but could mask issues during testing — always test with an empty honeypot field.
- `formLoadTime` is sent from the client and compared against `Date.now()` on the server. Clock skew between client and server could cause legitimate fast submissions to be silently dropped, or slow bot submissions to pass. The 500ms threshold is generous enough to avoid most false positives.
- The email fire-and-forget pattern means there's no retry logic. If Resend is down, the confirmation email is lost. Consider adding a background job queue for email delivery if reliability is critical.
- The `useForceLightTheme()` hook directly manipulates the DOM (`document.documentElement.classList`). If React's reconciliation or another component also modifies the class list, race conditions can occur. This pattern is acceptable for a standalone landing page but fragile in a deeply nested component tree.
- The `partner_applications` table has no index on `email` or `status`. If you need to query by email (e.g., checking for duplicate applications) or by status (e.g., listing pending reviews), add indexes.
