# Task Assignment System — QA & Product Review

## Executive Summary

The task assignment system in FridayReport.AI provides a solid foundation with many-to-many resource assignments, allocation percentages, automatic effort calculation, and multi-view consistency. All high-severity and medium-severity issues have been fixed. Low-priority feature gaps (auto-assignment, inline Gantt assignment) are deferred.

---

## 1. Assignment Logic

### How It Works
- Tasks use a many-to-many junction table (`task_resource_assignments`) linking tasks to resources with an `allocationPercentage` (default 100%).
- The `PUT /api/tasks/:taskId/resources` endpoint uses a **replace-all** strategy: delete all existing assignments, then insert the new set.
- Assignment changes trigger automatic recalculation of `estimatedHours` based on resource capacity and allocation.
- Notifications are sent to newly assigned and newly unassigned resources.

### Findings

| # | Finding | Severity | Status | Details |
|---|---------|----------|--------|---------|
| 1.1 | **No validation of resource IDs before assignment** | **High** | **FIXED** | Resource IDs are validated before insertion. Non-existent, cross-org, or soft-deleted IDs return a clear 400 error. (`resourceStorage.ts`) |
| 1.2 | **Replace-all strategy loses concurrent edits** | **Medium** | **FIXED** | The assignment endpoint now supports optimistic locking via `expectedUpdatedAt`. The client sends the task's last-known `updatedAt` timestamp; the server compares it against the current value and returns 409 Conflict if stale. (`resourceRoutes.ts`, `use-resources.ts`, call sites) |
| 1.3 | **No automatic assignment capabilities** | **Low** | Deferred | All assignments are manual. No auto-assignment based on workload, skills, or availability. This is a feature gap, not a bug. |

## 2. Scheduling Accuracy

### How It Works
- Dates use a working-days model (Mon–Fri, 8h/day) with `calculateEndDate` and `calculateDuration` utilities.
- The `propagateScheduleForProject` engine performs topological sort-based dependency propagation with support for FS/SS/FF/SF dependency types and lag days.
- Critical Path Method (CPM) calculates float and identifies critical tasks.
- Dates are stored as ISO strings (`YYYY-MM-DD`) without time components.

### Findings

| # | Finding | Severity | Status | Details |
|---|---------|----------|--------|---------|
| 2.1 | **No double-booking or overlap detection for resources** | **Medium** | **FIXED** | Over-allocation is now detected at assignment time (see 4.1) and warnings are displayed via toasts. Full date-level double-booking detection remains a future enhancement. |
| 2.2 | **Timezone handling is implicit** | **Low** | **FIXED** | Organizations now have a `timezone` field (default UTC) configurable in General Settings. This provides an explicit timezone reference per organization. (`shared/schema.ts`, `GeneralSection.tsx`, `organizationRoutes.ts`) |
| 2.3 | **Dependency propagation is robust** | — (Positive) | — | The topological sort engine correctly handles all four dependency types, detects cycles, and auto-adjusts downstream tasks. |

## 4. Availability & Workload Management

### How It Works
- Resources have `weeklyCapacity` (default 40h) and `availability` percentage fields.
- The `resource_availability` table tracks time off (PTO, sick, holiday) with date ranges.
- Utilization is calculated as: `(total allocation % / 100) × weeklyCapacity ÷ effective capacity × 100`.
- Over-allocation is flagged when total allocation exceeds 100%.
- An AI-powered optimization service (`resourceOptimizationAI.ts`) analyzes workloads and suggests rebalancing.

### Findings

| # | Finding | Severity | Status | Details |
|---|---------|----------|--------|---------|
| 4.1 | **Over-allocation is detected but not prevented** | **Medium** | **FIXED** | The assignment endpoint checks total allocation per resource and returns warnings when allocation exceeds 100%. Warnings are surfaced as toast notifications in the UI. (`resourceRoutes.ts`, all call sites) |
| 4.2 | **Time-off not checked during assignment** | **Medium** | **FIXED** | The assignment endpoint cross-references the `resource_availability` table and warns about time-off conflicts. (`resourceRoutes.ts`, all call sites) |
| 4.3 | **Utilization dashboard provides good visibility** | — (Positive) | — | The CapacityPlanningView and ResourceUtilizationDashboard surface clear over-allocation indicators. |

## 5. Notifications & Updates

### How It Works
- In-app notifications are created via `notificationEngine.ts` and stored in the `notifications` table.
- Assignment notifications are triggered when resources are newly assigned.
- Unassignment notifications are sent when resources are removed from a task.
- Task field change notifications are sent to assigned resources when key fields (status, dates, priority) change.
- Scheduled batch checks (cron) detect overdue tasks, upcoming deadlines, project health issues, and approaching milestones.
- Deduplication prevents repeat notifications within a 1-hour window for assignments/field changes and 24 hours for status alerts.

### Findings

| # | Finding | Severity | Status | Details |
|---|---------|----------|--------|---------|
| 5.1 | **No notification on unassignment** | **Medium** | **FIXED** | `createTaskUnassignmentNotification` function added; route compares old vs. new resource sets and sends unassignment notifications. (`notificationEngine.ts`, `resourceRoutes.ts`) |
| 5.2 | **No notification on reassignment to existing assignees** | **Low** | **FIXED** | A `createTaskFieldChangeNotification` function was added. When key task fields (status, startDate, endDate, priority) change, all assigned resources (except the person making the change) are notified. 1-hour deduplication prevents flooding. (`notificationEngine.ts`, `projectItemRoutes.ts`) |
| 5.3 | **Deduplication works correctly** | — (Positive) | — | The `notificationExists` check prevents flooding users with duplicate alerts. |

## 6. Data Integrity

### How It Works
- All views (Table, Gantt, Kanban) read from the same API endpoints and use TanStack Query for caching.
- Cache invalidation on mutation ensures cross-view consistency without page reload.
- Task deletion and resource deletion both use transactions to cascade-clean the junction table.

### Findings

| # | Finding | Severity | Status | Details |
|---|---------|----------|--------|---------|
| 6.1 | **Assignment update is not wrapped in a transaction** | **High** | **FIXED** | `updateTaskResourceAssignments` wraps the delete-then-insert in a `db.transaction()`. (`resourceStorage.ts`) |
| 6.2 | **Cross-view consistency is solid** | — (Positive) | — | TanStack Query cache invalidation ensures Kanban, Table, and Gantt views all reflect changes immediately. Mutation hooks now also invalidate org-level and project-level assignment query keys for Calendar/grouping views. |
| 6.3 | **Soft-delete inconsistency for resources** | **Medium** | **FIXED** | All three `getTaskResourceAssignments` query variants filter out soft-deleted resources (`isNull(resources.deletedAt)`). (`resourceStorage.ts`) |

## 8. Permissions & Access Control

### How It Works
- Organization-level roles: `owner`, `org_admin`, `team_member`, `super_admin`, `member`.
- `team_member` users are filtered to see only tasks they're assigned to or projects they're invited to.
- Email verification is required for write operations.
- Billing/credit limits gate task creation.

### Findings

| # | Finding | Severity | Status | Details |
|---|---------|----------|--------|---------|
| 8.1 | **Assignment endpoint lacks explicit role check** | **High** | **FIXED** | The `PUT /api/tasks/:taskId/resources` endpoint checks: authentication, org access, role (blocks `team_member`), and email verification. (`resourceRoutes.ts`) |
| 8.2 | **Read-only mode is now enforced server-side for assignments** | **Medium** | **FIXED** | The role check (8.1 fix) ensures `team_member` users cannot modify assignments even if they bypass the UI. |
| 8.3 | **Project-level assignment permissions are absent** | **Medium** | **FIXED** | `member` role users now have their project access verified using `getTeamMemberProjectIds` before being allowed to modify assignments. Users without access to the task's project receive a 403 error. (`resourceRoutes.ts`) |
