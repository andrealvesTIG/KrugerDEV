Requirements:

Input: tasks { id, name, start?, end?, durationDays, type(milestone?), } and dependencies { fromTaskId, toTaskId, type: FS|SS|FF|SF, lagDays }.

Build a dependency DAG (topologically sort). If a cycle is detected, stop and show a validation error listing the tasks in the cycle.

Convert milestone tasks to durationDays = 0.

Algorithm:

Forward pass (earliest dates):

Set ES (earliest start) for tasks with no predecessors to project start (or their fixed start if “pinned”).

For each task in topo order, compute candidate ES from each predecessor based on dependency type + lag:

FS: ES = max(ES, pred.EF + lag)

SS: ES = max(ES, pred.ES + lag)

FF: EF = max(EF, pred.EF + lag) then ES = EF - duration

SF: EF = max(EF, pred.ES + lag) then ES = EF - duration

EF = ES + durationDays.

Define project finish = max EF across tasks (or a selected “finish milestone” if provided).

Backward pass (latest dates):

Initialize terminal tasks’ LF (latest finish) = project finish.

Traverse tasks in reverse topo order and compute constraints from successors:

FS: LF = min(LF, succ.LS - lag)

SS: LS = min(LS, succ.LS - lag) then LF = LS + duration

FF: LF = min(LF, succ.LF - lag)

SF: LS = min(LS, succ.LF - lag) then LF = LS + duration

LS = LF - durationDays.

Compute Total Float: TF = LS - ES (or LF - EF). A task is critical if TF == 0 (use small tolerance, e.g. <= 0.0001).

Compute Critical Path chain:

From project start, pick critical tasks connected by dependencies where the relationship is “tight” (successor ES matches predecessor constraint) to produce an ordered critical path list.

UI / Output:

Store CPM fields on each task: ES, EF, LS, LF, TF, isCritical.

Highlight critical tasks/links in the Gantt (e.g., red bar/outline) and optionally show a “Show Critical Path” toggle.

Recalculate CPM whenever tasks, durations, lags, or links change (debounced).

Edge cases:

Multiple starting/ending tasks.

Mixed dependency types and lags.

Tasks with fixed/pinned dates (treat as constraints; if impossible, surface conflict).

Calendar handling: assume working days for now; keep date math centralized so calendars can be added later.
