Act as a senior QA engineer, backend engineer, and database architect. Audit this entire project end to end and fix issues systematically.

Your objectives:
1. Run a full QA pass across the codebase.
2. Compare the expected database schema with the actual implemented schema.
3. Identify missing database objects.
4. Detect and fix overall project issues, including broken logic, integration issues, and structural inconsistencies.

Detailed tasks:

A. QA and Application Audit
- Review the full codebase for runtime errors, broken flows, missing validations, unhandled exceptions, dead code, and inconsistent naming.
- Check frontend, backend, API routes, auth flows, forms, data fetching, error handling, and environment variable usage.
- Find any incomplete features, placeholder logic, TODOs, mock implementations, or fake data that should be replaced or clearly isolated.
- Verify CRUD flows work correctly.
- Verify loading, success, empty, and error states are handled properly.
- Check logs for failures and trace root causes.
- Produce fixes directly in code where safe and necessary.

B. Database Schema Comparison
- Inspect the application models, ORM definitions, migrations, SQL files, repository logic, and queries.
- Compare:
  - ORM/entity/model definitions
  - migration files
  - seed files
  - actual SQL usage in code
  - foreign keys, indexes, constraints, defaults, enums, and nullable fields
- Identify mismatches between code expectations and the actual database structure.
- Report missing columns, wrong types, missing constraints, orphaned relations, duplicated tables, and naming inconsistencies.
- Fix the mismatches safely.

C. Missing Database Objects
- Detect any missing:
  - tables
  - columns
  - indexes
  - foreign keys
  - views
  - triggers
  - functions/procedures
  - enums/types
  - seed data required for app startup
- Create the missing objects using the project’s existing migration strategy.
- Do not make destructive changes unless clearly required and safe.
- Preserve backward compatibility where possible.

D. Overall Fixes
- Fix import issues, broken references, wrong file paths, inconsistent types, and invalid assumptions between modules.
- Repair any failing API contracts between frontend and backend.
- Ensure validation schemas match database and API expectations.
- Normalize inconsistent naming between UI, API, services, and DB.
- Improve resilience, especially around null handling, async behavior, retries, and edge cases.

E. Output Requirements
- First produce a concise audit summary with:
  1. QA issues found
  2. schema mismatches
  3. missing DB objects
  4. code fixes applied
  5. remaining risks or manual review items
- Then implement the fixes.
- Show all changed files.
- For database changes, generate proper migration files.
- For every fix, explain:
  - root cause
  - change made
  - impact
- If something is ambiguous, choose the safest assumption and document it.

Execution rules:
- Be thorough, not superficial.
- Prefer root-cause fixes over temporary patches.
- Do not remove working functionality.
- Keep code style consistent with the existing project.
- Validate after changes by rerunning checks/tests/build if available.
- At the end, provide a final status report with:
  - fixed
  - partially fixed
  - not fixed
  - recommended next steps

Start by scanning the entire project structure and identifying the stack, database technology, ORM/migration tooling, and the highest-risk areas.
