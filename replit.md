# FridayReport.AI - Project Portfolio Management Application

## Overview
FridayReport.AI is an enterprise-grade project portfolio management application designed for comprehensive oversight of projects, portfolios, risks, key dates, and issues. It aims to enhance decision-making and operational efficiency through robust tracking, clean data presentation, and a professional user interface. The project's vision is to become a leading tool in project portfolio management.

## User Preferences
Preferred communication style: Simple, everyday language.

## System Architecture

### UI/UX Decisions
The application features an enterprise-grade UI inspired by Linear and Asana, utilizing a page-based architecture with reusable UI components. Styling is managed with Tailwind CSS and shadcn/ui. Data visualizations are powered by Recharts, and animations by Framer Motion. Industry-specific landing pages are responsive and SEO-optimized. The PMO Radar visualization uses HTML Canvas for dynamic, animated displays supporting both light and dark themes.

### Technical Implementations
**Frontend**: Developed with React 18 and TypeScript, employing Wouter for routing, TanStack React Query for state management, React Hook Form with Zod for form handling, and `@tanstack/react-virtual` for virtual scrolling.
**Backend**: Built with Express.js and TypeScript, utilizing PostgreSQL with Drizzle ORM. It provides a RESTful API with typed routes and Express sessions for session management.
**Shared Code**: Common schemas, route definitions, and models are shared between frontend and backend.
**Data Model**: Supports Portfolios, Projects, hierarchical Tasks, Resources, Risks, Issues, Portfolio Key Dates, Change Requests, Project Documents, and Organizations. It includes multi-tenancy with soft-delete and role-based access control. Task Milestones are integrated into the tasks table, while Portfolio Key Dates are managed in a dedicated table with various types and statuses.
**Security**: Features soft-delete, mandatory email verification, bot protection, versioned user consent tracking, encrypted OAuth tokens, `helmet` HTTP security headers, and `express-rate-limit` on all auth endpoints. Password policy requires 8+ characters with uppercase, lowercase, and digit. Investor Room password stored as environment variable (no hardcoded fallback). All API endpoints enforce authentication and authorization with role-based access checks on write operations.
**Notification Engine**: Supports various notification types with severity levels and deduplication. Granular per-user notification preferences cover ~50 notification keys (email + in-app channels) grouped into 8 categories (Account, Organizations, Projects, Tasks, AI, Timesheets, Marketing, Support). Catalog lives in `shared/notificationCatalog.ts` with helpers `resolvePreference`, `sanitizePreferenceUpdate`, `isAllEmailDisabled`, `isRequiredNotification`. Preferences stored as JSONB on `users.notification_preferences`. Required transactional rows (sign-in, password reset, magic link, email verification, organization invite, resource invite, passwordless sign-in) are locked on. Server-side gating: `shouldSendNotification` wraps `storage.createNotification` (returns null when blocked); `shouldSendEmailToAddress` is invoked inside every optional sender helper in `server/services/email.ts` and at every direct `sendEmail` call site (project status reports, meeting minutes/agendas, scheduled reports, AI agent emails, timesheet submission/approval/escalation/digest, partner application confirmations, intake/access-request/welcome/upgrade emails). Master "all email" switch disables every non-required email at once. Profile → Notifications page renders grouped per-row email/in-app toggles backed by `GET/PUT /api/profile/notification-preferences`.
**Help & Feedback**: Provides a system for user support tickets with text and screenshots.
**Friday Academy Training & Certification**: Offers interactive training modules with progress tracking and PDF certificate generation, including quiz mechanics.
**User Guide**: Comprehensive in-app documentation lives in `client/src/pages/UserGuide.tsx` (authenticated, 46 sections covering all platform features including the Construction Suite, Gantt/Timeline, PMO Radar, Power BI Agent, Templates, and Training) with a downloadable PDF generated via `@react-pdf/renderer`. A public-facing mirror in `client/src/pages/PublicUserGuide.tsx` (36 sections, dark theme) is served at `/guide` for SEO and prospect education. Both use hash-based section navigation; the PDF exports as Version 2.0 dated May 2026 with TOC auto-generated from the sections array.
**Integration Settings**: Stores organization-scoped integration credentials and connection statuses with encrypted OAuth tokens.
**API Design**: Follows a consistent API error convention and maintains a comprehensive OpenAPI 3.0 specification with extensive documentation for endpoints, schemas, and tags, including requiredness annotations and pagination. **OpenAPI schemas are auto-generated from Drizzle ORM table definitions** via `server/openapi-schema-generator.ts` and `server/openapi-schemas.ts`, ensuring the API documentation stays coupled to the database schema. The Drizzle tables are the single source of truth; OpenAPI property overrides (descriptions, enums, examples) are layered on top. **API documentation uses `@asteasolutions/zod-to-openapi` + `swagger-ui-express`**: `server/route-registry.ts` (108 lines) provides `apiRoute()` which co-locates Express handler + OpenAPI metadata in each route file, plus lightweight helpers (pathId, body, ref, r200, etc.). `server/swagger.ts` (71 lines) uses `OpenApiGeneratorV3` to build the spec document and merges in collected paths + auto-generated schemas. All route files use `apiRoute()`. 516 documented endpoints across 397 paths with 56 schemas. Swagger UI at `/api-docs`, raw spec at `/api-docs.json`.
**Performance Optimizations**: Includes extensive database indexing, N+1 query fixes, `React.memo` and `useCallback` for UI components, bulk assignment fetching, virtual scrolling, memoized date parsing, server-side date filtering, and optimized batch SQL updates.
**Code Organization**: Critical oversized files are modularized, duplicate industry landing pages consolidated, and Vite `manualChunks` configured for vendor code splitting. Unused code is systematically removed.
**Testing**: Utilizes Vitest for testing, focusing on date calculation logic, performance, and cross-validation.

**Friday Agent (Jarvis)**: AI-powered project assistant accessible via floating orb button. Backend service queries organization's project data (projects, tasks, issues, resources, dependencies) and feeds context to OpenAI for intelligent responses. Uses OpenAI function/tool calling to actually execute write actions (create tasks, risks, issues, add notes, flag projects) when the user confirms — actions are executed server-side via database operations, not just described in text. Features include voice input/output via Web Speech API, "Hey Friday" wake word detection, concise mode toggle, clickable entity links in responses, and sessionStorage-persisted chat history. **Page-scoped context**: Agent detects the current page (project/portfolio/resource) via URL path and sends `pageContext` to the server, which injects focused entity data into the system prompt so the agent prioritizes the entity the user is currently viewing. Context-aware suggested prompts change based on the current page. **File attachments**: Users can attach up to 5 files (500KB each) to agent messages — text-based files (CSV, JSON, TXT, etc.) are base64-decoded and included in the AI prompt; a paperclip button and file preview chips are shown in the input area. **CSV task import**: When a CSV file is attached, the agent parses it, maps flexible column names (e.g. "Task Name", "Owner", "Due Date") to task fields, presents a summary, and uses `bulk_create_tasks` tool to create up to 200 tasks at once in the target project. **Task updates**: `update_task` tool allows changing status, priority, assignee, dates, progress, description, milestone/critical flags on any task. Includes status/progress auto-sync (completing sets 100%, progress > 0 sets In Progress). **Resource assignment**: `assign_resources_to_task` tool assigns organization resources to tasks via the `task_resource_assignments` join table, supporting allocation percentages and validating resource IDs belong to the org. **Resource creation**: `create_resource` and `bulk_create_resources` tools allow the agent to add new resources (people/team members) to the organization with fields like name, email, title, department, skills, experience level, hourly rate, availability, etc. These are org-scoped tools that don't require a project ID. Files: `server/services/jarvisService.ts`, `server/routes/jarvisRoutes.ts`, `client/src/hooks/use-jarvis.ts`, `client/src/hooks/use-speech.ts`, `client/src/hooks/use-wake-word.ts`, `client/src/components/jarvis/JarvisOrb.tsx`, `client/src/components/jarvis/JarvisPanel.tsx`.

**Database Migration & Deployment**: The build process includes schema synchronization and additional index/manual migrations.
**Schema Integrity**: A script validates all tables, columns, and foreign keys.
**Schema Drift Detection**: `server/schemaDriftCheck.ts` runs at server boot (before routes accept traffic) and compares the Drizzle schema (`shared/schema.ts` plus models in `shared/models/*.ts`) against the live database. Any missing tables or columns are logged loudly to stderr in a banner. In development the boot is aborted via `process.exit(1)` so drift cannot be ignored; in production the check logs but keeps serving so partial functionality stays online. Set `SKIP_SCHEMA_DRIFT_CHECK=true` to bypass. **Workflow for schema changes**: (1) edit `shared/schema.ts`, (2) run `npm run db:push` to sync the database, (3) restart the dev server — if you forget step 2 the boot fails immediately with a list of the missing tables/columns. Use `npm run db:check-schema` for a deeper diff that also covers types, nullability, defaults, and foreign keys.
